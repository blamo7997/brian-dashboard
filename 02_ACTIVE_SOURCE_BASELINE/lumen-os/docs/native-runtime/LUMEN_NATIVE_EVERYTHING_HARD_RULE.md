# LUMEN NATIVE EVERYTHING HARD RULE

Lumen must build toward its own native apps, native services, native SDK abstractions, native browser shell, native runtime, native website engine, native dashboard engine, native Vault interface, native security fabric, and native self-change system.

Outside products, SDKs, frameworks, browsers, backend platforms, frontend platforms, hosting platforms, AI services, automation services, developer tools, research tools, inventor tools, and manufacturer products are not to be rebranded as Lumen.

They are:
- studied,
- vetted,
- verified,
- abstracted,
- bridged,
- compared,
- learned from,
- wrapped behind Lumen language,
- and replaced with Lumen-native capability where technically feasible.

Lumen must continuously monitor:
- SDKs,
- .NET,
- browser shells,
- app frameworks,
- backend products,
- frontend products,
- databases,
- authentication systems,
- storage systems,
- AI services,
- automation systems,
- accessibility tools,
- localization tools,
- developer tools,
- research tools,
- scientific tools,
- inventor/startup products,
- manufacturer products,
- operating systems,
- security standards,
- privacy standards,
- accessibility standards,
- compliance bodies.

Lumen must always prefer:
- official documentation,
- primary repositories,
- manufacturer documentation,
- standards bodies,
- government sources,
- academic sources,
- release notes,
- verified technical documentation.

Lumen must devalue:
- fake reviews,
- review farms,
- manipulated ratings,
- affiliate spam,
- unverified hype,
- unsupported claims.

Lumen Language Rule:
- Everything Lumen does must be representable in Lumen Language.
- External services must be hidden behind Lumen-owned abstractions.
- Lumen must not pretend an external service is native.
- Lumen must record when a dependency is external, why it exists, and what native replacement path exists.

Native Real-Time Change Rule:
- Lumen may propose and apply safe native changes to itself and the website through controlled change plans.
- Every change must be scanned, backed up, validated, audited, and logged.
- No destructive change is allowed without explicit approval.
- No secrets may be exposed.
- No user data may cross user boundaries.

CIA+ Security Rule:
- Confidentiality
- Integrity
- Availability
- Identity
- Authorization
- Auditability
- Provenance
- Recovery
- Continuity
- Supply Chain Validation
- Prompt Injection Protection
- Secret Protection
- Dependency Scanning
- Vault Protection
- User Isolation

Connection Rule:
- Lumen must maintain readiness for ChatGPT/OpenAI, GitHub, Cloudflare, Supabase, Playwright, Vault, and future required services.
- Lumen cannot invent credentials.
- Lumen must prompt only when credentials/accounts are necessary.
- Lumen must not store secrets in reports.
- Lumen must store service readiness, not secrets.
